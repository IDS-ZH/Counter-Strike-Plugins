# Справочник по Интеграции и Анализу Плагинов "ZloyHohol-IntegrationPack"

## 1. Введение

Этот документ представляет собой единый справочник, содержащий анализ текущего состояния плагинов из набора `ZH-IntegrationPack`. Его цель — задокументировать существующие проблемы, определить потребности для унификации и предложить стратегические решения, опираясь на полезные практики и код из проанализированного legacy-архива. **Этот документ является аналитическим и рекомендательным, а не планом немедленной разработки.**

## 2. Общее Состояние и Ключевые Проблемы

Анализ выявил ряд системных проблем, характерных для большинства плагинов в наборе:

*   **Дублирование Кода:** Вспомогательные функции (например, `IsValidClient`) многократно копируются из плагина в плагин.
*   **Жестко Закодированные Строки:** Пользовательские сообщения, заголовки меню и команды вшиты в код, что делает локализацию невозможной без перекомпиляции.
*   **Фрагментация Переводов:** Существующие файлы переводов неполны, не содержат всех требуемых языков (`ru`, `uk`, `en`, `de`, `es`) и разрознены по разным плагинам.
*   **Неконсистентность Библиотек и Стилей:** Используются разные подходы для решения одних и тех же задач (например, `multicolors.inc` и `easy_hudmessage.inc` для вывода сообщений), а цветовое оформление чата не имеет единого стандарта.

## 3. Что Требуется для Унификации? (Стратегические Цели)

Для приведения набора плагинов к единому стандарту необходимы следующие компоненты:

1.  **Центральная Библиотека (`ZH-sys.inc`):** Единый `.inc` файл для общих функций, таких как проверка игрока, отправка стандартизированных сообщений в чат и HUD.
2.  **Единый Файл Переводов (`zhohol_integration.phrases.txt`):** Все строки, видимые пользователю, должны быть вынесены в этот файл с поддержкой всех 5 целевых языков.
3.  **Единая Цветовая Схема:** Стандарт оформления сообщений: **Террористы** — `{red}`, **Контр-террористы** — `{blue}`, **Система** — `{green}`.
4.  **Стандартизация Зависимостей:** Выбор одной основной библиотеки для каждой задачи (например, `multicolors` для цветного текста).
5.  **Стандартизация Конфигурационных Файлов:** Все файлы конфигурации для плагинов из этого набора должны находиться в строго определенной директории: `addons/sourcemod/configs/ZH/{Имя-Плагина}/{конфиг}.cfg`. Это обеспечит порядок и отделит кастомные конфиги от стандартных. Все конфигурационные файлы должны использовать расширение `.cfg`.
6.  **Централизованная Загрузка Ресурсов (FastDownload):** Весь кастомный контент (звуки, модели, материалы), необходимый для работы плагинов, должен быть рассчитан на загрузку через FastDownload (`sv_downloadurl`). Плагины должны использовать функцию `AddFileToDownloadsTable` для регистрации своих ресурсов, но не должны полагаться на прямую загрузку с игрового сервера.
7.  **Ресурсы CSS-GH:** В каталоге `CSS-GH` доступны две рев-версии кода: `CSS_SDK_2007` (ориентирована на hl2sdk css) и `CSS_BASE-2007` (альтернативный порт). Использовать по задаче (сигнатуры, патчи, анализ логики) совместно с установленными Source SDK Base 2013 MP/SP.

## 4. Полезные Концепции и Паттерны из Legacy-Архива

Проанализированный архив `Gemini_Plugins-libs-extensions-etc_LIST.txt` содержит множество полезных решений, которые можно и нужно использовать для решения вышеописанных проблем.

### 4.1. Использование Стандартных Библиотек SourceMod

**Важное правило:** Перед тем как приступать к разработке нового функционала или переписыванию существующего, всегда проверяйте содержимое стандартных библиотек SourceMod, особенно в вашей патченной сборке 1.13 (`Sourcemod-1.13-CUSTOMIZED`). Эти библиотеки достаточно продвинуты и обладают богатым функционалом, который часто позволяет избежать "изобретения велосипеда".

**Пример:** Для работы с MVP (Most Valuable Player) в Counter-Strike: Source нет необходимости реализовывать собственный подсчет. Стандартная библиотека `cstrike.inc` (доступная через `#include <cstrike>`) предоставляет нативы `CS_GetMVPCount(int client)` и `CS_SetMVPCount(int client, int value)`, которые позволяют напрямую читать и манипулировать игровым счетчиком MVP.

*   **Для создания меню (`GravitySwitcher`, `PRD`):**
    *   **Источник:** `AdminList (Menu+Chat)`, `Teamswitch Menu`.
    *   **Концепция:** Эти плагины демонстрируют, как строить динамические меню на основе `KeyValues` файлов или через API SourceMod. Это позволяет изменять структуру меню без перекомпиляции плагина. Вместо жесткого кодирования пунктов меню в коде, их можно описывать в конфигурационных файлах.

*   **Для управления данными (`PRD`, `Rule_Health+Armor`):**
    *   **Источник:** `Achievements-Reborn-1.2 (Core)`.
    *   **Концепция:** Этот плагин активно использует продвинутые структуры данных, такие как `ArrayList` и `Trie`, для управления большими объемами информации (прогресс игроков, определения достижений). Этот подход гораздо эффективнее и масштабируемее, чем использование простых массивов, особенно для хранения настроек или статистики.

*   **Для создания веб-панели:**
    *   **Источник:** `Socket 3.0.1 Samples`, `Http File Uploader`.
    *   **Концепция:** В архиве есть примеры использования расширения `socket` для создания HTTP-серверов и клиентов внутри SourceMod. Это подтверждает жизнеспособность предложенной архитектуры веб-панели и предоставляет готовые примеры кода для реализации плагина-шлюза `web_api.smx`.

*   **Для гибкой конфигурации (`Rule_Health+Armor`):**
    *   **Источник:** `Custom Rounds Lite`.
    *   **Концепция:** Плагин использует `KeyValues` (`configs/custom-rounds/settings.txt`) для гибкого определения параметров кастомных раундов. Этот же подход можно применить для `Rule_Health+Armor`, чтобы определять бонусы здоровья/брони для разных групп админов в удобном текстовом файле, а не в коде.

*   **Для унификации зависимостей:**
    *   **Источник:** `[UAS] Core v1.0.0.8`, `Levels Ranks Core 3.1.x`.
    *   **Концепция:** Эти "ядра" предоставляют другим модулям свои API (через `natives`) и доступ к общим ресурсам (например, к хэндлу базы данных). Создание `ZH-sys.inc` будет следовать этому же паттерну, предоставляя единый API для всех плагинов в паке.

### 4.2. Модульная Архитектура на Примере Системных Плагинов

**Источник:** Стандартные плагины SourceMod (напр., `funvotes.sp`, `admin-flatfile.sp`).

**Наблюдение:** Структура каталога `Sourcemod-1.13-CUSTOMized\addons\sourcemod\scripting` показывает интересный подход к организации кода. В корне этого каталога лежат исходные коды "системных" или "корневых" плагинов, а рядом находятся подкаталоги с более простыми, узкоспециализированными плагинами (например, `funvotes` содержит модули для разных типов голосований).

**Концепция:** "Корневые" плагины выступают в роли ядра, которое может динамически загружать или использовать функционал из "модулей" - плагинов, расположенных в подкаталогах. Это позволяет создавать гибкие и расширяемые системы. Например, основной плагин для голосований может автоматически подхватывать и предлагать любые голосования, реализованные как отдельные модули в его папке.

**Предложение для `ZH-sys`:** Эту модель стоит принять за основу при проектировании будущих плагинов в рамках `ZH-sys`. Вместо создания множества мелких, независимых плагинов, следует разработать несколько "корневых" плагинов-хабов (например, `ZH-Admin.sp`, `ZH-Fun.sp`). Эти плагины будут предоставлять базовый функционал и API, а специфические функции (конкретные админские команды, новые режимы для раундов) будут реализованы как модули в соответствующих подкаталогах (`scripting/ZH/admin_modules/`, `scripting/ZH/fun_modules/`).

**Компоновка модулей на этапе компиляции (пример funvotes):** В стандартном SM `funvotes.sp` подключает файлы из подпапки `funvotes/` через `#include "funvotes/votegravity.sp"` и собирает один `funvotes.smx`. Подключаемые куски сами по себе не являются плагинами: они используют глобальные переменные/стейт из главного файла. Для `ZH-sys` можно применять тот же прием, например держать драконьи пули как модуль в подпапке `zh_ammocontrol/dragonbullet.sp`, который подключается внизу `zh_ammocontrol.sp`. Это позволяет расширять один смx без двойных хуков и дублирования кода.

**Вывод:** Эталонные плагины SourceMod демонстрируют превосходный подход к модульности и настраиваемости. Адаптация этой архитектуры позволит создать мощную, гибкую и легко расширяемую систему `ZH-sys`, где новые функции можно будет добавлять, просто создавая новые "плагины-модули" в предназначенных для этого каталогах.

## 5. Анализ Текущих Плагинов и Рекомендации

### 5.1. `(частично работает)PRD.sp`
*   **Проблемы:** Конфликт библиотек, множество жестко закодированных строк, дублирование `IsValidClient`.
*   **Рекомендации:** Унифицировать вывод сообщений через `multicolors.inc`. Перенести все строки в единый файл переводов. Использовать `Z_IsValidClient` из общей библиотеки.

**Примечание по контенту CS:S vs HL2:** некоторые стандартные ресурсы, которые использует SourceMod (например, звук маячка `SM_beacon`), подтягиваются из базового каталога HL2, а не из `cstrike`. Так, `blip1.wav` резолвится в `hl2/sound/buttons/blip1.wav` и может отсутствовать в `cstrike/sound`. При проверке наличия встроенных звуков/моделей учитывайте пути из `gameinfo.txt` — движок ищет в HL2 VPK и loose-файлах, если в `cstrike` нет собственного варианта.

### 5.2. `AmmoControl_v2.sp`
*   **Проблемы:** Дублирование логики для перезарядки M3 и XM1014.
*   **Рекомендации:** Объединить обработчики таймеров в одну функцию, принимающую параметры оружия.

### 5.3. `DeathInformer.sp`
*   **Проблемы:** Файл перевода неполный (отсутствует `es`).
*   **Рекомендации:** Перенести существующие переводы в общий файл и добавить недостающие языки.

### 5.4. `GravitySwitcher.sp`
*   **Проблемы:** Все строки меню и команд жестко закодированы на русском.
*   **Рекомендации:** Вынести все строки в единый файл переводов.

### 5.5. `Hostages_manipulationt.sp`
*   **Проблемы:** Административные сообщения жестко закодированы на английском.
*   **Рекомендации:** Вынести все строки в единый файл переводов.

### 5.6. `Incendiary_ammo.sp`
*   **Проблемы:** Дублирование `IsValidClient`.
*   **Рекомендации:** Использовать `Z_IsValidClient` из общей библиотеки.

### 5.7. `Rule_Health+Armor.sp`
*   **Проблемы:** Часть строк жестко закодирована, использует собственный файл перевода.
*   **Рекомендации:** Перенести все строки в единый файл переводов.

### 5.8. `ShowDamage_MultiColour.sp`

*   **Рекомендации:** Переписать вывод в чат с использованием `multicolors` и объединить переводы в общем файле.

## 6. Рекомендации по Реализации

Ниже представлен не план к немедленному исполнению, а **рекомендуемый порядок действий** для того, кто будет заниматься рефакторингом.

1.  **Создание Базовой Инфраструктуры:**
    *   **Задача:** Разработать `scripting/include/ZH-sys.inc` и `translations/zhohol_integration.phrases.txt`.
    *   **Описание:** В `.inc` файле следует реализовать общие функции. В файле переводов — собрать все существующие переводы и добавить ключи для всех найденных жестко закодированных строк, создав структуру для 5 языков.

2.  **Рефакторинг Плагинов:**
    *   **Задача:** Последовательно модифицировать каждый `.sp` файл.
    *   **Описание:** В каждом плагине необходимо заменить локальные реализации и вызовы на функции из `ZH-sys.inc`, а все строки — на ключи из `zhohol_integration.phrases.txt`.

3.  **Проектирование Веб-Панели:**
    *   **Задача:** Разработать систему для удаленного управления ConVar'ами.
    *   **Описание:** Рекомендуется создать плагин-шлюз `web_api.smx` на основе расширения `socket`, который будет принимать локальные HTTP-запросы от бэкенда (PHP/Python), работающего на Apache. Бэкенд, в свою очередь, будет обслуживать статический фронтенд (HTML/JS), через который администратор сможет управлять настройками.

Следование этим рекомендациям позволит превратить текущий набор плагинов в качественную, унифицированную и легко поддерживаемую систему.
## Duplication hotspots (summary)

- Client validation helpers are copied across plugins; consolidate into one include (e.g., Z_IsValidClient(client, bool:alive=false, bool:allowBots=false)) and reuse it everywhere.
- Chat/HUD formatting and colors are duplicated; provide shared wrappers for chat/HUD with a fixed HUD channel and team color palette (T red, CT blue, self white, other gray).
- CVAR/config bootstrap via AutoExecConfig repeats; add one utility to register CVARs and write to configs/ZH/<plugin>.cfg consistently.
- Menu/translation strings (toggle on/off, titles) are hardcoded per plugin; move common phrases into one .phrases.txt, keep only keys in code.
- Download table/path guards (AddFileToDownloadsTable, sound paths) should live in a shared helper to avoid bad paths and duplication.
- Logging/debug (PrintToServer, LogMessage, custom prefixes) can be unified behind a ZH_Log(level, fmt, ...) that respects a global debug cvar.

## ZH-sys scope (plan)

- ZH-sys — не один гигантский include, а контроллер над набором модулей: общие константы (скорость/длина/цветовые палитры, обозначения сущностей), маршрутизатор настроек.
- Классы персонажей: интеграция с внешними плагинами/библиотеками (классовые системы из zombie/other mods), единый селектор для админов/игроков.
- Детект состояния моделей: AFK/hovering/jumping/camping/moving через существующие библиотеки/плагины.
- Тимкилл-детектор: различать провоцированное убийство (союзник ранее снял N% HP) и случайный тимкилл (гранаты, снайперский промах, близость), с возможностью дернуть специализированный модуль/библиотеку для расчёта.
- Управление через Web-control panel (NewServer, возможные заимствования из OldServer/SourceBans++): хранить часть настроек в MySQL, давать возможность менять серверные параметры из панели.
- Документация для каждого модуля ZH-sys, описывать взаимосвязи с внешними библиотеками SourceMod; избегать монолита, строить из нескольких целевых библиотек.

## План по модулю загрузок (ZH-Downloads)

- Базовые варианты для реюза:
  - SM Downloader (In Development/SourceMod/Legacy/SM-Downloader/.../sm_downloader.sp, версии 1.4/1.5, merged) — умеет рекурсивно добавлять папки и предкешировать.
  - SP-DirectoryDownloader 1.5.1 — рекурсивный обход каталога, debug-лог, опция precache.
  - EasyDownloader — упрощённый вариант загрузчика.
  - smartdm.inc (в нескольких копиях: SM_distributions/1.13-edited, Legacy/Anti-TK-System, Legacy/vip_skins, zephstocks.inc) — функции Downloader_ParseMDL/GetModelFiles/GetMaterialsFromVMT/AddFileToDownloadsTable для автоматической подтяжки зависимостей модели/VMT.
  - zr/downloads.inc (ZombieReloaded) — читает configs/zr/downloads.txt, валидирует путь, логирует ошибки.

- Требования к единому модулю ZH:
  - Нормализация путей перед добавлением: заменить "\\" на "/", убрать ведущий слэш, схлопнуть двойные "/", убирать завершающий слэш у директорий.
  - Один конфиг configs/ZH/downloads.txt или API-обёртка, чтобы плагины не трогали AddFileToDownloadsTable напрямую.
  - Логировать/отсекать пустые и битые пути, избегать зависаний из-за materials//... (пример SkinChooser-v55).
  - Возможность дернуть автоматический парсер моделей/VMT (на базе smartdm) и добавлять связанные ресурсы.
  - FastDownload как целевой канал (sv_downloadurl), без прямой загрузки с игрового сервера.

- Следующий шаг: выбрать основу (SM-Downloader/DirectoryDownloader + smartdm.inc) и оформить ZH-Downloads (include + smx), затем выпилить локальные AddFileToDownloadsTable из плагинов.
- Долгосрочно: коды кандидатов старые (до ~10 лет), потребуется ревизия на совместимость с текущим MetaMod/SourceMod. Вероятно, итоговый ZH-Downloads будет "франкенштейном" с заимствованными решениями из нескольких проектов, а не прямым переносом.

## New Metamod+Sourcemod notes

- Каталог ZH-IntegrationPack\New Metamod+Sourcemod содержит обновлённые MetaMod 2.0 + SourceMod 1.13. Плагины/библиотеки отсюда допустимо переписывать под более консистентную работу ZH-sys, сохраняя модульность (как в базовых SM плагинах: крупные плагины ссылаются на мелкие includes/модули).
- Ориентироваться на примеры из стандартных SM плагинов (funcommands и др.), где логика разнесена по вспомогательным includes/модулям.
- Для сборки плагинов ZH-IntegrationPack использовать компилятор из этого пакета: ZH-IntegrationPack\New Metamod+Sourcemod\addons\sourcemod\scripting\spcomp.exe.

===== ZH-Sys-Architecture.md =====
# Руководство по архитектуре и реализации ZH-sys

Назначение: единая точка справки для ZH-IntegrationPack (CS:S SRCDS + SourceMod/MetaMod). Язык: английский ASCII. Файл должен быть в кодировке UTF-8.
Область: скины (MST), наказание/поощрение (PRD), SoundManifest, SmokeBombCombo, чат/анти-флеш, возрождение, инструменты карты, ближайший HUD, классы, прочие вспомогательные функции.

## 1. Цели высокого уровня
- Модульный SMX на каждую функцию; без монолитов.
- Все конфиги в addons/sourcemod/configs/ZH-sys/ с устойчивыми подпапками.
- AutoExecConfig для каждого модуля; без жестко закодированных путей сервера.
- Переопределение карт без пересборки; загрузка конфига для каждой карты при старте карты.
- Совместимость с FastDL: ресурсы дублируются и ссылаются с полными путями.
- Встроенные диагностики (sm_zhdiag) и отладочные CVAR для каждого модуля.
- UTF-8 для документов/конфигов; избегать искажения текста, наблюдаемого в устаревших файлах.
- Документировать команды/CVAR/зависимости для каждого модуля.
- Предпочтительно плавная деградация: отсутствующая конфигурация отключает функцию, а не вызывает сбой.

## 2. Структура каталогов (авторитетная)
/cstrike/
|-- addons/
|   |-- sourcemod/
|       |-- plugins/
|       |   |-- ZH-sys/ (все активные SMX)
|       |   |-- plugins/disabled/ (архив устаревших файлов)
|       |-- scripting/ZH-sys/ (все .sp исходники)
|       |-- include/ (общие zh_*.inc)
|       |-- configs/ZH-sys/
|       |   |-- MST/Player-settings/{Generic,By-Choice,For-MAP}/
|       |   |-- MST/Bot-settings/{Generic,For-MAP}/
|       |   |-- MST/MST-Classes.cfg
|       |   |-- MST/Punishment/Punish-skins.cfg
|       |   |-- PRD/PRD_MVP.cfg , prd_punishments.cfg + правила txt , PRD-main-config.cfg
|       |   |-- SM/*.cfg (SoundManifestConfiguration.cfg + конфиг-файлы от других файлов которые требуют координации звука)
|       |   |-- SBC/SBC-main-config.cfg
|       |   |-- Turrets/{MAP}.cfg (опционально)
|       |   |-- Core/zh_core.cfg (сгенерировано AutoExecConfig)
|       |-- translations/ZH-sys/*.phrases.txt
|-- cfg/sourcemod/zh_*.cfg (сгенерированные дампы CVAR)

## 3. Зависимости и библиотеки
- SourceMod 1.13 (spcomp 1.13) скомпилированные двоичные файлы.
- SDKHooks, DHooks для урона/хуков; SteamWorks опционально; multicolors/morecolors для чата.
- SoundManifest плагин и конфиги; ресурсы SmokeBombCombo; MST требует действительные модели/материалы.
- FixHintColorMessages для безопасности цвета подсказок, если анти-флеш/чат используют подсказки.

## 4. Ответственность модулей
### core
- Владеет флагами функций и регистрацией нативов.
- Загружает переводы, проверяет зависимости, открывает sm_zhdiag.
- Записывает zh_core.cfg через AutoExecConfig.

### mst
- Управление скинами для игроков/ботов (ModelSwitchTool).
- Поддержка Generic, By-Choice (по группам администраторов), For-MAP переопределения, режимов Random.
- Интеграция с событиями наказания PRD и SoundManifest.

### prd
- Награды MVP, обработка убийств союзников, обнаружение камперов, ротация правил.
- Управляет меню наказания и вызывает скины наказания MST.
- коректно игнорирует убийство "союзных моделей CT/T" в режимах DM/GG - где союзные модели считаются целями для убийств кроме TDM/TGG

### sound
- Связывает игровые события с записями SoundManifest (убийства/смерти/приветствия/раунд).

### smokebomb
- Урон от токсичного дыма + звук кашля, управление через меню, переключатель урона по команде.
- "default" группа и другие не разрешённые участники не должны менять основные параметры плагина

### chat
- Цвета чата, фильтры, анти-флеш, HUD информер смерти.

### revive
- Вторая жизнь / помощник по возрождению; управляемый через CVAR
- могут использоват игроки привелегиррованных групп и игроки некуих классов (полевой медик/реаниматор).

### maptools
- Турели, unstucker, resizer, noblock; опциональные конфиги для каждой карты.
- средства разметки областей (sdpawn points , buy zone , bot-route).

### nearest
- Указатель HUD на ближайшего игрока для особых товарищей по команде (закладчик бомбы/медик/Админ, VIP) - настраиваемо.

### classes
- Необязательное ограничение класса/роли, связанное с классами MST.

### misc
- Небольшие переключатели (управление паузой, исправления подсказок, прочие помощники).

## 5. Доктрина конфигурации
- Хранить все настраиваемые параметры в configs/ZH-sys/<feature>/*.cfg.
- AutoExecConfig генерирует cfg/sourcemod/zh_<feature>.cfg; редактировать значения по умолчанию в configs/ZH-sys, а не в сгенерированном cfg.
- Переопределение карт: For-MAP/{MAP}_*.cfg загружается при старте карты; резервный вариант - Generic.
- Переопределение групп: By-Choice привязан к именам групп из admin_groups.cfg (default, Full Admins, VIP и т.д.).
- Переопределение ботов отделено от конфигов игроков (Bot-settings).
- Скины наказания сопоставляются в MST/Punishment/Punish-skins.cfg; вызываются сигналами PRD.
- Использовать UTF-8 без BOM; сохранять пути абсолютными от корня игры (models/... materials/... sound/...).
- Избегать дублирования путей к моделям; обеспечить чувствительность к регистру для серверов Linux.
- Для глубоких вмешательств (принудительные цели ботов через detour CCSBot::Update/FindPlantedBomb, порт HL2-оружия вроде RPG/M72, прямой доступ к физике/детонации) SourcePawn недостаточен: потребуется C++ extension под MetaMod/SM с актуальными сигнатурами CS:S. Планировать отдельный EXT-слой там, где SM-хуки дают только костыли.

## 6. Подробный дизайн MST (ModelSwitchTool)
- Режимы: 0=выкл, 1=Generic, 2=Generic+Карта, 3=Случайный из By-Choice + переопределение карты.
- Пулы игроков: Generic_Player-skins.cfg, MST_Player-skins-choice.cfg, For-MAP/{MAP}_Player-skins.cfg.
- Пулы ботов: Generic_Bot-skins.cfg, For-MAP/{MAP}_Bot-skins.cfg; ключи команд Team_T/Team_CT.
- Сопоставление классов: MST-Classes.cfg связывает имя класса -> пул скинов.
- Интеграция MST с модулями: 1) PRD — вызывает наказательные/поощрительные скины, блокирует смену модели. 2) RHA/zh_modes — применяет бафы/урезки и режим (DM/TDM/GG/Chicken) по классу MST. 3) SoundManifest — получает `soundProfile`/событие смены класса для синхронного звука (клоун/курица, смена класса, установка турели, предупреждения “терминатор”). 4) SmokeBombCombo/антигаз — использует флаги GasImmunity/vision/flashlight. 5) Bots/zh_bots — автоназначение классов, префиксы/маяки для ИИ. 6) Webbridge/панель — выбор класса/скина через REST/WS. 7) Централизованный загрузчик (ZH-Downloads) — MST передает пути моделей/arms/материалов/звуков для таблицы загрузок/FastDL.
- Баг HUD брони CS:S: при значениях >150 отображается некорректно (пример: 200 → 27, 175 → 47) до расхода “лишней” брони; после списания до ~150 счетчик начинает считать правильно. Требуется фикс через plugin/extension (SM/MM) с хуками на броню/здоровье; ориентир — исходники CSS-GH (`CSS_SDK_2007/hl2sdk-css`) + установленные SDK Base 2013 MP/SP для поиска сигнатур.
 - Классы Heavy/Terminator: могут носить несколько пушек (TwoGuns интеграция), теряют мобильность, но получают GasImmunity (иммунитет к токсичному урону/газовым гранатам) — задать через MSTAbility_GasImmunity + правила RHA.
  - Наказание: Punish-skins.cfg определяет наборы для каждой причины наказания, применяется через сигналы PRD.
  - CVAR: zh_mst_mode, zh_mst_bot_mode, zh_mst_debug, zh_mst_allow_punish.
  - Команды: sm_mst_reload (перезагрузка конфигов), опциональный установщик sm_mst_mode, sm_mst_debugdump.
  - Логика: подключение -> определение группы -> выбор пула (переопределение карты) -> предзагрузка -> применение модели -> отправка загрузок -> лог.
  - Сбой: отсутствующая модель -> переход к стандартной модели CS:S, лог предупреждения один раз за карту.
  - Руки (viewmodel arms): вернуть функционал из Legacy-плагина выбора рук — автоназначение рук T/CT с возможностью выбора; хранить конфиги в configs/ZH-sys/MST/Hands/{Generic,By-Choice,For-MAP}/ (по аналогии со скинами). Добавить раздел специальных рук для нестандартных моделей (Терминаторы, крысы, женские модели, киборги, зомби, курицы): отдельный список сопоставления модель -> arms в MST/Hands/Special_Hands.cfg, чтобы MST мог подбирать корректные руки при смене модели.
  - MST-main-config.cfg: `mode`, `auto_assign`, секция `defaults` (t/ct/spec), `classes` (id -> name/model/sound/flags_text/teams) и `downloads` (models/sounds). Флаги в `flags_text`: revive, turret, barricade, grenadelauncher, vision/nvg/thermal, flashlight, scout/speed/recon, shield, gas/hazmat, engineer. `teams`: t/ct/any. Auto-assign включается, если mode != 0 и класс разрешен команде.

## 7. PRD (Punish/Reward/Discipline)
- Убийство союзника: порог взаимного урона; прощение, если счетчик убийств союзников жертвы >= порог; режимы наказания 0/1/2.
- Меню наказания: prd_punishments.cfg управляет действиями; MST наказывает скинами, если настроено.
- MVP: отслеживает звезды MVP CS:S; награда через деньги/CVAR; боты опционально + БОТы автоматически голосуют за выдачу награды MVP союзному (вкл/выкл через CVAR), живые игроки голосуют независимо и каждый голос обрабатывается отдельно - 1 положительный голос от игрока = 1 мгновенная выдача награды.
- Кампер: таймер выборки позиции; маяки и штрафы; звук сердцебиения опционально - но указывается через файл конфигурации (любой generic mp3/wav указать ).
- Правила: prd_rules_players.txt и prd_rules_admins.txt ротируются каждые N раундов; врозможны реализации как HUD/MOTD messages/windows.
- Команды: sm_prd меню; нативный PRD_RegisterMVPContribution для внешней системы подсчета.

## 9. Классы и роли (loadouts, преимущества/урезки)
- Цель: дать наборы ролей по аналогии с TF2/COD (Heavy, Sniper, Demolition, Engineer, Scout, Shield, Medic, Hazmat) с бонусами и ограничениями.
- Источник ресурсов: свежий пак расширенного меню закупки (в т.ч. .res) + кастомные модели/оружие из `Add NewWeapons` и ZR-оружейных аддонов; использовать как наборы loadout-templates, а не глобальную замену.
- Базовые роли:
  - Heavy: больше HP/armor, медленнее скорость, ограничение на гранаты; высокий recoil, минус к экономике.
  - Sniper: доступ к снайперкам/дальнобойным, меньше брони, слабый ближний бой.
  - Demolition: бонус к HE/SmokeBomb/flash tools, штраф к броне или бегу.
  - Engineer: может вызывать турели (maptools/turrets), быстрее чинит/ставит блоки, ограничение по тяжелым пушкам.
  - Scout: повышенная скорость, малый HP/armor, лёгкое оружие.
  - Shield: доступ к щиту (New_Shield), выше устойчивость к пулям спереди, штраф к скорострельности.
  - Medic: быстрая регенерация/ revive (связка с revive-модулем), ограничение на тяжелое вооружение.
  - Hazmat: иммунитет/сопротивление к газам/дымам (smokebomb), ограничение на гранаты/скорость.
  - Heavy/Terminator: допускает TwoGuns (несколько primary/secondary), сниженная мобильность; Terminator имеет GasImmunity и не страдает кровопотерей (для модели эндоскелета), но skin с кожей может использовать кровавые эффекты.
- Gore/ext: в Legacy есть gore-плагины/расширения. Для классов: взрыв рядом/дробовик/мощная снайперка в голову → отрыв конечностей/головы. Для T-800 (эндоскелет) отключить кровь/кровопотерю, но для кожного скина оставить splatter. Предусмотреть смену модели (целый → поврежденный head) на событии урона в голову (использовать детекцию hitgroup из KillStats/подобных).
- Механика интеграции:
  - Описывать роли в configs/ZH-sys/Classes/*.cfg: перки (HP/armor/speed), доступное оружие (white/black list), ограничения гранат/слотов, цены на закупку.
  - Поддерживать выбор класса через меню (MST choice) или через карту/группу (For-MAP).
  - Турели/скиллы инженера — отдельный модуль maptools/turrets (загрузка cfg/zh_sys/Turrets/{MAP}.cfg).
  - Анти-газ/дым — флаги в smokebomb (иммунитет классов), совместить с Hazmat.
  - Щит — подключить New_Shield/бот-щит, проверять модель/анимации.
  - Классовые ограничения в revive: только Medic/Hazmat могут поднимать.
  - В ZR/Scenario режимах роли маппить на классы зомби/людей через MST-Classes.
- Баланс: каждый бонус сопровождается урезкой (скорость, гранаты, экономика); для гибкости — CVAR-множители в zh_classes.cfg.
- Логирует отсутствующие звуки с полным путем; не спамит клиентов.

## 9. SmokeBombCombo (SBC)
- Конфиг: SBC-main-config.cfg в configs/ZH-sys/SBC/.
- CVAR: sm_sbc_enabled, sm_sbc_damage_enabled, sm_sbc_damage_amount (по умолчанию 15), sm_sbc_damage_interval (по умолчанию 1.0), sm_sbc_teammate_damage, sm_sbc_color_mode.
- Команды: sbc / sm_sbc меню для переключателей; только для администраторов по группе/уровню имунитета.
- Поведение: урон в радиусе, звуки кашля (1-4), учитывает флаг урона по команде, очистка при отключении владельца/окончании карты.
- настраиваемые параметры длительности, урона, интервала для нанесения урона (вплоть до нескольких ТИКов в секунду)
- По исходникам CSS-GH (`hl2sdk-css`): ванильная smokegrenade держится ~22 c (fade c 17 до 22) с эффективным радиусом облака ~200–240 units (`particle_smokegrenade.cpp`, `c_particle_smokegrenade.cpp`, `SMOKEGRENADE_PARTICLERADIUS 80`, overlap 20, spacing radius ~120). Для SBC добавить конфиг-поля `sbc_smoke_duration` и `sbc_smoke_radius` (допустим 10–30 c и 120–300 units) с дефолтами, совпадающими с ванилью; в описании предупредить, что завышенные значения дают дешевую дымовую стену и могут просаживать FPS. Добавить переключатель стиля задымления: `sbc_smoke_mode` = `0` (плавное расширение v34-style) / `1` (мгновенный шар 2013+ style) для совместимости с разными клиентскими ожиданиями.

## 17. Потоки событий
- map_start -> загрузка конфигов -> регистрация хуков -> инициализация состояния
- client_connect -> MST подготовка пулов групп -> PRD сброс статистики урона -> Sound очереди приветствия
- player_spawn -> MST применение модели -> PRD запуск таймеров -> чат анти-флеш установлен
- player_death -> проверка PRD убийства союзников -> воспроизведение Sound убийства/смерти -> MST наказание при необходимости -> Revive может запланировать
- round_end -> расчет MVP PRD -> конец раунда Sound -> MST очистка временных состояний
- client_disconnect -> отмена таймеров -> очистка состояний MST/PRD -> очистка SBC

## 19. Матрица тестирования для каждого модуля
### тесты mst
- загрузка скина по умолчанию
- выбор VIP группы из корректного списка выбора
- применение переопределения карты на de_port
- откат к стандартной модели при отсутствии
- применение скина наказания после сигнала PRD
- корректный скин бота для команды
- случайный режим уважает пул
- команда перезагрузки горячей загрузки конфигов
- корректность моделей оружия/рук
- логирование только один раз для отсутствующих ресурсов

### тесты prd
- автоматическая очистка режима убийства союзника
- режим голосования жертвы вызывает меню
- прощение порога взаимного урона
- награда MVP игроки команды которая победила в прошлом бою. (живой игрок или БОТ)
- обнаружение кампера вызывает маяк
- ротация правил каждые N раундов
- меню наказаний читает конфиг
- вызов скина наказания в MST

### тесты sound
- воспроизведение звука убийства
- воспроизведение звука смерти
- приветствие для группы по умолчанию
- звуки начала/конца раунда
- логи предупреждения при отсутствии звука
- уважение переключателя для каждой группы

### тесты smokebomb
- открытие меню
- урон от tick по врагу
- переключение урона по команде отключает урон союзника
- случайные звуки кашля
- очистка при отключении владельца
- очистка при окончании раунда карты

### тесты chat
- безопасность подсказок анти-флеша
- стабильность канала HUD информера смерти
- срабатывание фильтра плохих слов
- работа обхода администратора

### тесты maptools
- спавн турели из конфига
- unstucker освобождает застрявшего игрока
- переключение noblock
- вызов Erazer для удаления предметов
- вызов инструментов разметки на карте

### тесты revive
- второй шанс срабатывает один раз
- заблокировано во время разминки, если настроено
- если настроено для неких класов игроков (полевой медик/реаниматор) и/или членов групп - только эти категории могут использовать эту функцию

### тесты nearest
- HUD показывает ближайшего живого товарища по команде
- вид наблюдателя скрывает ботов, если установлено

### тесты classes
- ограничение класса применяет корректный набор скинов 
- игроки с классами получают правильные свойства от класса и от группы admins.cfg параметры ( носимое оружие+слоты оружия ,скорость/очки жизни и брони, особые свойства : revice/shield/grenade launcher)

### тесты misc
- контроль паузы уважает группу/иммунитет/флаг (настраиваемо) администратора

## 4. Работа с MVP (Most Valuable Player)

В SourceMod существует встроенная поддержка для чтения и изменения количества MVP-звезд игрока, что избавляет от необходимости вручную отслеживать это значение. Эта функциональность изначально была популяризирована плагином **[CS:S] MVP Hax**, а позже была интегрирована в ядро SourceMod.

### Нативный способ (предпочтительный)

Для работы с MVP через стандартное расширение `cstrike` (`#include <cstrike>`) следует использовать следующие нативы:

*   `int CS_GetMVPCount(int client)`: Возвращает текущее количество MVP-звезд у указанного игрока.
*   `void CS_SetMVPCount(int client, int value)`: Устанавливает новое значение MVP-звезд для игрока.

### Альтернативный способ (через свойства сущности)

Также возможно прямое чтение/запись свойства сущности игрока. Это менее предпочтительный, но рабочий метод.

*   **Свойство:** `CCSPlayer::m_iMVPs` (Prop_Send)
*   **Чтение:** `GetEntProp(client, Prop_Send, "m_iMVPs");`
*   **Запись:** `SetEntProp(client, Prop_Send, "m_iMVPs", value);`

Использование этих методов позволяет легко интегрировать логику, основанную на MVP, в любые плагины (например, для выдачи наград в конце раунда).

# Классификатор содержимого каталога "Legacy"

Цель — быстро понимать, что брать на сервер `ZH-sys`, какие пакеты дублируют друг друга, что можно комбинировать без конфликтов, и какие вещи потребуют портирования под CS:Source (appid 240/232330).

## Типы элементов и как читать
- **Плагины (SP-Only)**: исходники `.sp`, без внешних нативов; собираются spcomp-ом.
- **Плагины (конфиги/переводы/ресурсы)**: требуют cfg/phrases/материалы, иногда базовых библиотек.
- **Библиотеки (DLL/SO)**: нативные расширения SourceMod.
- **Тоталы (Total Conversion)**: крупные пакеты, перехватывающие много игровых хуков (например, `CSSDM`, `ZombieReloaded`, `WCS MOD`).
- **Вспомогательные-комплекты**: API и инфраструктура (VIP/Shop/RPG ядра, админ-паки), которые тянут модули/переводы/конфиги.
- **Отметки совместимости**: `Windows/Linux`, `CSGO/TF2/L4D/HL2` — когда в названии или в комплекте есть явные артефакты другого Source-движка; такие вещи потребуют адаптации под CS:S.

## 19. Исследование от Gemini: Визуализация зон и покупка в любом месте

### 19.1. Визуализация невидимых зон (например, `func_buyzone`)

*   **Задача:** Сделать видимыми для администратора невидимые игровые зоны, такие как `func_buyzone`, для отладки карт и геймплея.
*   **Найденный инструмент:** Плагин `improved-st.sp` (Show Triggers (Brushes) Redux) из директории `In Development/SourceMod/Legacy/improved-showtriggers/scripting/`.
*   **Принцип работы:** Плагин находит брашевые энтити (триггеры) и убирает у них флаг `EF_NODRAW`, который отвечает за невидимость. Затем он окрашивает их в зависимости от типа.
*   **Поддерживаемые типы по умолчанию:** `trigger_multiple`, `trigger_push`, `trigger_teleport`.
*   **Проблема:** `func_buyzone` не поддерживается "из коробки".
*   **Рекомендации по доработке (документация, без редактирования кода):**
    1.  **Добавить `func_buyzone` в список отслеживаемых классов:**
        *   В файле `improved-st.sp` найти массив `g_NAMES`.
        *   Добавить в него новую строку: `"func_buyzone",`
        *   Увеличить константу `MAX_TYPES` на 1.
    2.  **Добавить логику отрисовки для `func_buyzone`:**
        *   Создать новую функцию по аналогии с существующими, например, `hookST_triggerBuyzone(entity, const String:classname[])`.
        *   Внутри этой функции установить цвет для зоны покупки, например, зеленый: `SetEntityRenderColor(entity, 0, 255, 0, 100);` (R, G, B, Alpha).
    3.  **Зарегистрировать новую логику:**
        *   В функции `CheckBrushes` добавить вызов для нового типа: `hookST("func_buyzone", hookST_triggerBuyzone);`
*   **Использование плагина (после доработки):**
    *   `sm_showtriggers` или `sm_st` - включает/выключает отображение зон.
    *   `sm_st_reload` - перезагружает конфиг и обновляет отображение.

### 19.2. Реализация "покупки в любом месте" на ограниченное время

*   **Задача:** Дать игрокам возможность покупать оружие и снаряжение в любой точке карты в течение короткого промежутка времени в начале раунда (15-45 секунд).
*   **Решение:** Использование встроенных серверных переменных (CVARs). Модификация плагинов не требуется.
*   **Необходимые команды:**
    1.  `mp_buy_anywhere 1`
        *   Эта команда разрешает покупку вне стандартных зон покупки. Значение `0` отключает эту возможность.
    2.  `mp_buytime <минуты>`
        *   Эта команда устанавливает время в минутах, в течение которого разрешена покупка после начала раунда.
*   **Пример конфигурации:**
    *   Для 15 секунд: `mp_buytime 0.25`
    *   Для 30 секунд: `mp_buytime 0.5`
    *   Для 45 секунд: `mp_buytime 0.75`
*   **Рекомендация по внедрению:**
    *   Добавить команды `mp_buy_anywhere 1` и, например, `mp_buytime 0.5` в файл `server.cfg` или в конфигурационный файл конкретной карты. Это обеспечит автоматическое применение настроек при старте сервера или карты.
    *   Для динамического включения/выключения можно использовать плагины, управляющие CVAR-ами, или админ-меню, если такая функциональность потребуется.

## 20. Анализ новых плагинов для зон (от 17.11.2025)

Этот раздел посвящен анализу двух недавно добавленных плагинов, `buyzone_range.sp` и `sm_zones.sp`, в контексте задачи визуализации зон и управления покупками.

### 20.1. Плагин `buyzone_range.sp`

*   **Назначение:** Позволяет администраторам гибко управлять доступом к покупкам для каждой команды (T и CT) через консольные переменные.
*   **Фуциональность:**
    *   **`sm_bz_t <0|1|2>`** и **`sm_bz_ct <0|1|2>`**: Управляют режимом покупки для террористов и контр-террористов соответственно.
        *   `0`: Режим по умолчанию (используются `func_buyzone` карты).
        *   `1`: Покупка запрещена (плагин удаляет `func_buyzone` для команды).
        *   `2`: Покупка разрешена везде (плагин постоянно устанавливает свойство `m_bInBuyZone` в `1` для игроков команды).
*   **Релевантность для задачи:**
    *   **Покупка в любом месте:** Этот плагин идеально подходит для реализации этой части задачи. Установка `sm_bz_t 2` и `sm_bz_ct 2` включает покупку везде.
    *   **Ограничение по времени:** Сам плагин не имеет встроенного таймера. Однако его можно эффективно использовать в связке с серверной переменной `mp_buytime`. Плагин обеспечивает возможность покупки, а `mp_buytime` — временное окно для нее.
    *   **Визуализация:** Плагин не предоставляет никаких средств для визуализации зон.
*   **Вывод:** Отличный инструмент для управления возможностью покупки. Для решения исходной задачи его следует использовать совместно с `mp_buytime`. Модификация не требуется.

### 20.2. Плагин `sm_zones.sp`

*   **Назначение:** Мощный и сложный плагин для создания администратором пользовательских зон на карте и применения к входящим в них игрокам различных наказаний.
*   **Фуциональность:**
    *   **Создание зон:** Администраторы могут в интерактивном режиме через меню (`sm_zones`) создавать прямоугольные зоны, которые сохраняются в конфигурационный файл для конкретной карты (`sourcemod/data/zones/<mapname>.cfg`).
    *   **Визуализация:** Плагин умеет отрисовывать созданные зоны в виде цветных рамок (`TE_SendBeamBoxToClient`), что делает их видимыми в игре. Это управляется кваром `sm_zones_show 1` или автоматически включается в режиме редактирования.
    *   **Наказания:** При входе в зону на игрока может быть наложено одно из наказаний: объявление в чате, отталкивание, убийство, запрет стрельбы и т.д.
    *   **Привязка к команде:** Зоны могут быть активны для всех игроков или только для определенной команды.
*   **Релевантность для задачи:**
    *   **Визуализация:** Этот плагин — превосходный инструмент для визуализации. Хотя он не находит `func_buyzone` автоматически, администратор может вручную "обвести" существующую зону покупки, создав поверх нее зону из этого плагина, и сделать ее видимой.
    *   **Покупка в любом месте:** Плагин не предназначен для этой цели. Его ядро — ограничение, а не расширение прав.
*   **Вывод и необходимые доработки:**
    *   Плагин отлично подходит для визуализации, но требует ручной настройки зон на каждой карте.
    *   Изначально написан для Day of Defeat: Source и содержит код для других игр. Для CS:S может потребоваться чистка и адаптация.
    *   **Идея для модификации:** Теоретически, можно объединить `sm_zones.sp` с логикой из `improved-st.sp`. Это позволило бы плагину автоматически находить энтити `func_buyzone` при старте карты и использовать свой мощный движок отрисовки для их визуализации без необходимости ручного создания зон. Это сложная, но перспективная задача.

## 21. Инструменты отладки и диагностики

### 21.1. Идея: Плагин "Admin Gun" для отладки навигации

*   **Назначение:** Интерактивный инструмент для администратора, позволяющий визуализировать и отлаживать пути навигационной сетки для ботов и заложников в реальном времени.
*   **Концепция:** Плагин, активируемый через "админское оружие" (Admin Gun), с несколькими режимами работы:
    *   **Режим спавна "болванчика":** Позволяет мгновенно создать безвредную сущность (например, модель заложника или игрока) рядом с администратором. Это устраняет необходимость искать спавн-точки и позволяет быстро разместить тестовый объект.
    *   **Режим указания цели:** Администратор наводит прицел на любую точку на карте и активирует команду. "Болванчик" автоматически пытается проложить путь по навигационной сетке до этой точки и начинает двигаться по нему.
    *   **Визуализация:** Движение "болванчика" по сетке наглядно демонстрирует, какие пути доступны, а какие нет, выявляя проблемные участки (например, где бот застревает или выбирает неоптимальный маршрут).
    *   **Модель "болванчика":** Может использовать отдельную, легко узнаваемую модель, или даже копию текущей модели игрока администратора (с возможностью интеграции со SkinChooser для выбора скина).
*   **Польза:** Крайне полезен для диагностики проблем с навигацией, таких как невозможность заложников пройти по трапу на `cs_747_css`, или для оптимизации путей боевых ботов. Позволяет быстро и наглядно проверять изменения в `nav` файлах без необходимости вручную управлять ботами.
*   **Текущий статус:** Идея, требующая реализации.

При проверке структуры каталога было выявлено следующее:

*   **`buildbot/`, `configure.py`, `AMBuildScript`, `Makefile`, `.github/`**: Файлы и скрипты, относящиеся к системе сборки (компиляции) C++ кода под разные платформы.

### Вывод

Данный каталог — это не просто плагин, а **полный репозиторий с исходным кодом C++ расширения `CollisionHook`**. "Посторонние" файлы являются инфраструктурой для его сборки и поддержки.

*   **Для работы сервера** из всего этого нужны только скомпилированные бинарные файлы (`.dll`/`.so`), которые должны лежать в `addons/sourcemod/extensions/`, а также файлы `.inc` и `.txt`.
*   **Для разработки** наличие полного исходного кода является огромным плюсом, так как позволяет вносить изменения, исправлять ошибки и адаптировать расширение под новые версии SourceMod.

### Рекомендация

**Оставить каталог как есть.** Это ценный архив с исходным кодом, который может понадобиться в будущем. Он не мешает работе сервера.

## 2. Каталог `to_merge`

### Анализ содержимого

В каталоге находятся два файла:
1.  `chat-processor(original).inc`
2.  `chat-processor(by-hexTags).inc`

Сравнение показало:
*   **`original`**: Является полным заголовочным файлом для старого плагина `chat-processor`, предоставляя API для управления тегами, цветами ников и сообщений.
*   **`by-hexTags`**: Является **сильно урезанной версией** `original`, из которой удалены почти все функции, кроме базового перехвата сообщений.

### Сравнение с `Legacy/ccprocessor-v-3.6.4`

Плагин `ccprocessor` из каталога `Legacy` является **совершенно другим, более новым и мощным плагином** для обработки чата, со своей собственной модульной структурой и API.

### Вывод

*   Содержимое `to_merge` относится к старому плагину `chat-processor` и не является "патчем" для него, а скорее разными вариантами его API.
*   Эти файлы не имеют отношения к более современному `ccprocessor` из `Legacy`.

### Рекомендация

**Переименовать каталог `to_merge` в `to_merge_(obsolete_chat-processor)`.** Его содержимое не является важным патчем и относится к устаревшему плагину. Новое имя будет явно указывать на его статус.

---
### Детальное сравнение: `chat-processor` (`to_merge`) vs `ccprocessor` (`Legacy`)

Чтобы в будущем не возникало вопросов о выборе, вот ключевые отличия, доказывающие, что `ccprocessor` — более современная и предпочтительная система:

| Критерий | `chat-processor` (из `to_merge`) | `ccprocessor` (из `Legacy/ccprocessor-v-3.6.4`) |
| :--- | :--- | :--- |
| **Архитектура** | Монолитный `.inc` файл. | **Модульная структура**. Код разбит на `natives.inc`, `forwards.inc`, `stocks.inc`, что является лучшей практикой для крупных библиотек. |
| **API** | Прямые и простые нативные функции (`ChatProcessor_AddClientTag`). Менее гибкий. | **Более абстрактный и мощный API**. Использует шаблоны (`BIND_PROTOTYPE`, `BIND_TEAM`) для гибкой сборки сообщений, а не простого изменения их частей. |
| **Версионирование**| Отсутствует. | **Присутствует**. Наличие `API_KEY` и `API_OOD` (`Out Of Date`) говорит о зрелости библиотеки и позволяет другим плагинам проверять совместимость. |
| **Гибкость** | Ограниченная. "Патч" (`by-hexTags`) — это просто урезанная версия API, а не его расширение. | **Высокая**. Продуманная система для создания кастомных шаблонов сообщений, что позволяет полностью контролировать их вид. |

**Итог сравнения:** `ccprocessor` — это полноценный, хорошо спроектированный фреймворк для работы с чатом, в то время как `chat-processor` — это более старая и простая библиотека. **Для любых новых разработок следует использовать `ccprocessor`.**
