# Справочник по Интеграции и Анализу Плагинов "ZloyHohol-IntegrationPack"

## 1. Введение

Этот документ представляет собой единый справочник, содержащий анализ текущего состояния плагинов из набора `ZH-IntegrationPack`. Его цель — задокументировать существующие проблемы, определить потребности для унификации и предложить стратегические решения, опираясь на полезные практики и код из проанализированного legacy-архива. **Этот документ является аналитическим и рекомендательным, а не планом немедленной разработки.**

## 2. Общее Состояние и Ключевые Проблемы

Анализ выявил ряд системных проблем, характерных для большинства плагинов в наборе:

*   **Дублирование Кода:** Вспомогательные функции (например, `IsValidClient`) многократно копируются из плагина в плагин.
*   **Жестко Закодированные Строки:** Пользовательские сообщения, заголовки меню и команды вшиты в код, что делает локализацию невозможной без перекомпиляции.
*   **Фрагментация Переводов:** Существующие файлы переводов неполны, не содержат всех требуемых языков (`ru`, `uk`, `en`, `de`, `es`) и разрознены по разным плагинам.
*   **Неконсистентность Библиотек и Стилей:** Используются разные подходы для решения одних и тех же задач (например, `multicolors.inc` и `easy_hudmessage.inc` для вывода сообщений), а цветовое оформление чата не имеет единого стандарта.

## 3. Что Требуется для Унификации? (Стратегические Цели)

Для приведения набора плагинов к единому стандарту необходимы следующие компоненты:

1.  **Центральная Библиотека (`ZH-sys.inc`):** Единый `.inc` файл для общих функций, таких как проверка игрока, отправка стандартизированных сообщений в чат и HUD.
2.  **Единый Файл Переводов (`zhohol_integration.phrases.txt`):** Все строки, видимые пользователю, должны быть вынесены в этот файл с поддержкой всех 5 целевых языков.
3.  **Единая Цветовая Схема:** Стандарт оформления сообщений: **Террористы** — `{red}`, **Контр-террористы** — `{blue}`, **Система** — `{green}`.
4.  **Стандартизация Зависимостей:** Выбор одной основной библиотеки для каждой задачи (например, `multicolors` для цветного текста).
5.  **Стандартизация Конфигурационных Файлов:** Все файлы конфигурации для плагинов из этого набора должны находиться в строго определенной директории: `addons/sourcemod/configs/ZH/{Имя-Плагина}/{конфиг}.cfg`. Это обеспечит порядок и отделит кастомные конфиги от стандартных. Все конфигурационные файлы должны использовать расширение `.cfg`.
6.  **Централизованная Загрузка Ресурсов (FastDownload):** Весь кастомный контент (звуки, модели, материалы), необходимый для работы плагинов, должен быть рассчитан на загрузку через FastDownload (`sv_downloadurl`). Плагины должны использовать функцию `AddFileToDownloadsTable` для регистрации своих ресурсов, но не должны полагаться на прямую загрузку с игрового сервера.

## 4. Полезные Концепции и Паттерны из Legacy-Архива

Проанализированный архив `Gemini_Plugins-libs-extensions-etc_LIST.txt` содержит множество полезных решений, которые можно и нужно использовать для решения вышеописанных проблем.

### 4.1. Использование Стандартных Библиотек SourceMod

**Важное правило:** Перед тем как приступать к разработке нового функционала или переписыванию существующего, всегда проверяйте содержимое стандартных библиотек SourceMod, особенно в вашей патченной сборке 1.13 (`Sourcemod-1.13-CUSTOMIZED`). Эти библиотеки достаточно продвинуты и обладают богатым функционалом, который часто позволяет избежать "изобретения велосипеда".

**Пример:** Для работы с MVP (Most Valuable Player) в Counter-Strike: Source нет необходимости реализовывать собственный подсчет. Стандартная библиотека `cstrike.inc` (доступная через `#include <cstrike>`) предоставляет нативы `CS_GetMVPCount(int client)` и `CS_SetMVPCount(int client, int value)`, которые позволяют напрямую читать и манипулировать игровым счетчиком MVP.

*   **Для создания меню (`GravitySwitcher`, `PRD_Codex`):**
    *   **Источник:** `AdminList (Menu+Chat)`, `Teamswitch Menu`.
    *   **Концепция:** Эти плагины демонстрируют, как строить динамические меню на основе `KeyValues` файлов или через API SourceMod. Это позволяет изменять структуру меню без перекомпиляции плагина. Вместо жесткого кодирования пунктов меню в коде, их можно описывать в конфигурационных файлах.

*   **Для управления данными (`PRD_Codex`, `Rule_Health+Armor`):**
    *   **Источник:** `Achievements-Reborn-1.2 (Core)`.
    *   **Концепция:** Этот плагин активно использует продвинутые структуры данных, такие как `ArrayList` и `Trie`, для управления большими объемами информации (прогресс игроков, определения достижений). Этот подход гораздо эффективнее и масштабируемее, чем использование простых массивов, особенно для хранения настроек или статистики.

*   **Для создания веб-панели:**
    *   **Источник:** `Socket 3.0.1 Samples`, `Http File Uploader`.
    *   **Концепция:** В архиве есть примеры использования расширения `socket` для создания HTTP-серверов и клиентов внутри SourceMod. Это подтверждает жизнеспособность предложенной архитектуры веб-панели и предоставляет готовые примеры кода для реализации плагина-шлюза `web_api.smx`.

*   **Для гибкой конфигурации (`Rule_Health+Armor`):**
    *   **Источник:** `Custom Rounds Lite`.
    *   **Концепция:** Плагин использует `KeyValues` (`configs/custom-rounds/settings.txt`) для гибкого определения параметров кастомных раундов. Этот же подход можно применить для `Rule_Health+Armor`, чтобы определять бонусы здоровья/брони для разных групп админов в удобном текстовом файле, а не в коде.

*   **Для унификации зависимостей:**
    *   **Источник:** `[UAS] Core v1.0.0.8`, `Levels Ranks Core 3.1.x`.
    *   **Концепция:** Эти "ядра" предоставляют другим модулям свои API (через `natives`) и доступ к общим ресурсам (например, к хэндлу базы данных). Создание `ZH-sys.inc` будет следовать этому же паттерну, предоставляя единый API для всех плагинов в паке.

## 5. Анализ Текущих Плагинов и Рекомендации

### 5.1. `(частично работает)PRD_Codex.sp`
*   **Проблемы:** Конфликт библиотек, множество жестко закодированных строк, дублирование `IsValidClient`.
*   **Рекомендации:** Унифицировать вывод сообщений через `multicolors.inc`. Перенести все строки в единый файл переводов. Использовать `Z_IsValidClient` из общей библиотеки.

**Примечание по контенту CS:S vs HL2:** некоторые стандартные ресурсы, которые использует SourceMod (например, звук маячка `SM_beacon`), подтягиваются из базового каталога HL2, а не из `cstrike`. Так, `blip1.wav` резолвится в `hl2/sound/buttons/blip1.wav` и может отсутствовать в `cstrike/sound`. При проверке наличия встроенных звуков/моделей учитывайте пути из `gameinfo.txt` — движок ищет в HL2 VPK и loose-файлах, если в `cstrike` нет собственного варианта.

### 5.2. `AmmoControl_v2.sp`
*   **Проблемы:** Дублирование логики для перезарядки M3 и XM1014.
*   **Рекомендации:** Объединить обработчики таймеров в одну функцию, принимающую параметры оружия.

### 5.3. `DeathInformer.sp`
*   **Проблемы:** Файл перевода неполный (отсутствует `es`).
*   **Рекомендации:** Перенести существующие переводы в общий файл и добавить недостающие языки.

### 5.4. `GravitySwitcher.sp`
*   **Проблемы:** Все строки меню и команд жестко закодированы на русском.
*   **Рекомендации:** Вынести все строки в единый файл переводов.

### 5.5. `Hostages_manipulationt.sp`
*   **Проблемы:** Административные сообщения жестко закодированы на английском.
*   **Рекомендации:** Вынести все строки в единый файл переводов.

### 5.6. `Incendiary_ammo.sp`
*   **Проблемы:** Дублирование `IsValidClient`.
*   **Рекомендации:** Использовать `Z_IsValidClient` из общей библиотеки.

### 5.7. `Rule_Health+Armor.sp`
*   **Проблемы:** Часть строк жестко закодирована, использует собственный файл перевода.
*   **Рекомендации:** Перенести все строки в единый файл переводов.

### 5.8. `ShowDamage_MultiColour.sp`
*   **Проблемы:** Строки меню и HUD жестко закодированы на русском.
*   **Рекомендации:** Вынести все строки в единый файл переводов.

### 5.9. `ShowMySteamID.sp`
*   **Проблемы:** Использует устаревшие цветовые коды и отдельные файлы для каждого языка.
*   **Рекомендации:** Переписать вывод в чат с использованием `multicolors` и объединить переводы в общем файле.

### 5.10. `SmokeBombCombo-V3.5.sp`
*   **Проблемы:** Строки меню жестко закодированы, использует две разные библиотеки для вывода сообщений.
*   **Рекомендации:** Вынести строки в файл переводов и унифицировать вывод через `multicolors`.

## 6. Рекомендации по Реализации

Ниже представлен не план к немедленному исполнению, а **рекомендуемый порядок действий** для того, кто будет заниматься рефакторингом.

1.  **Создание Базовой Инфраструктуры:**
    *   **Задача:** Разработать `scripting/include/ZH-sys.inc` и `translations/zhohol_integration.phrases.txt`.
    *   **Описание:** В `.inc` файле следует реализовать общие функции. В файле переводов — собрать все существующие переводы и добавить ключи для всех найденных жестко закодированных строк, создав структуру для 5 языков.

2.  **Рефакторинг Плагинов:**
    *   **Задача:** Последовательно модифицировать каждый `.sp` файл.
    *   **Описание:** В каждом плагине необходимо заменить локальные реализации и вызовы на функции из `ZH-sys.inc`, а все строки — на ключи из `zhohol_integration.phrases.txt`.

3.  **Проектирование Веб-Панели:**
    *   **Задача:** Разработать систему для удаленного управления ConVar'ами.
    *   **Описание:** Рекомендуется создать плагин-шлюз `web_api.smx` на основе расширения `socket`, который будет принимать локальные HTTP-запросы от бэкенда (PHP/Python), работающего на Apache. Бэкенд, в свою очередь, будет обслуживать статический фронтенд (HTML/JS), через который администратор сможет управлять настройками.

Следование этим рекомендациям позволит превратить текущий набор плагинов в качественную, унифицированную и легко поддерживаемую систему.
